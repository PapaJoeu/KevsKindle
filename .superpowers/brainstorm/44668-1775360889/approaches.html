Three Approaches to Modularize

All keep it static — no build tools, no bundler. Pick the one that fits best.

A

Vanilla Split — JSON + JS + CSS

Files: index.html (shell), styles.css, app.js, data/links.json, data/config.json

How it works: index.html becomes a minimal shell (~30 lines). On load, app.js fetches the two JSON files and renders cards into the DOM. All styling lives in styles.css.

Pros

  • Zero dependencies — just files
  • Edit JSON to add/remove links or change copy
  • Kindle browser can still render it (JS is simple)
  • GitHub Pages serves it as-is

Cons

  • Extra HTTP requests on load (CSS + JS + 2 JSON)
  • Brief flash before JS renders cards
  • Kindle's slow browser feels the extra requests
B

Vanilla Split + Inline Data Fallback

Files: index.html (shell + embedded data), styles.css, app.js, data/links.json, data/config.json

How it works: Same as A, but the JSON data is also embedded in index.html as a <script> tag with a JS object. app.js uses the embedded data immediately, no fetch needed. The JSON files exist as the canonical "source of truth" you edit, and a simple script copies them into index.html before deploy.

Pros

  • Fast load — no extra fetch for data
  • Still edit JSON files as your workflow
  • Works even as file:/// if you ever need it

Cons

  • Requires a build/copy step (even if simple)
  • Data exists in two places — JSON + index.html
  • Adds friction to "just edit and push"
C

Pure Vanilla Split — No Build Step

Files: index.html (minimal shell), styles.css, app.js, data/links.json, data/config.json

How it works: Same file structure as A, but app.js shows a lightweight placeholder (the header + "Loading...") immediately via HTML in the shell, then fetches and renders. A loading state styled to match the page makes the fetch invisible on fast connections, and graceful on slow ones (Kindle).

Pros

  • Zero dependencies, zero build steps
  • Edit JSON, push, done
  • Loading state handles Kindle's slow browser gracefully
  • Clean separation — each file has one job

Cons

  • Still 4 HTTP requests on first load
  • Won't work as file:/// (but you don't need that)

My Recommendation: C

You're on GitHub Pages, so the extra requests are fast and cached. No build step means your workflow stays "edit JSON, git push, done." The loading state keeps Kindle users happy. Approach B adds a build step for a problem you don't have, and A is essentially C without the loading UX polish.
